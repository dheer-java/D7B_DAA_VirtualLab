# Chapter 1: Introduction

## 1.1 Objective

To implement and analyse the **N-Queen Problem** using a **Backtracking** approach, and to build an interactive virtual-lab web application that visually demonstrates the algorithmic process of placing queens, detecting conflicts, and backtracking upon failure.

## 1.2 Problem Statement

Given an **N × N** chessboard, place **N** queens such that:

- No two queens share the **same row**.
- No two queens share the **same column**.
- No two queens share the **same diagonal** (both upper-left and lower-left).
- **All N queens** must be placed on the board simultaneously.
- The algorithm must find **all possible valid solutions**.

This is a classic **constraint-satisfaction** problem and a textbook example of the **backtracking** paradigm.

## 1.3 Motivation

The N-Queen problem is one of the most studied combinatorial problems in computer science. It serves as an excellent teaching tool for:

- **Backtracking algorithms** — understanding how to build solutions incrementally and abandon partial solutions that violate constraints.
- **Recursion** — the algorithm naturally decomposes into sub-problems (place a queen in the current column, recurse for the next).
- **State-space search** — visualising how the algorithm explores a tree of possibilities and prunes invalid branches.

# Chapter 2: Concept, Algorithm, and Data Structures Used

## 2.1 The N-Queen Problem — Concept

The N-Queen problem asks: **How can N queens be placed on an N×N chessboard so that no two queens attack each other?**

A queen in chess can attack any piece that lies in the same **row**, **column**, or **diagonal**. Therefore, a valid placement requires that every pair of queens differs in row, column, and both diagonals.

### Historical Context

- The problem was first posed by chess composer **Max Bezzel** in 1848 for the 8-queen case.
- The first solutions were published by **Franz Nauck** in 1850.
- It has been widely studied in algorithms, AI, and constraint programming.

## 2.2 Backtracking — The Algorithmic Paradigm

**Backtracking** is a systematic method for generating all (or some) possible configurations of a search space. It is essentially a **depth-first search (DFS)** with **pruning**.

### Core Idea:

1. **Choose** — Pick an option for the current decision point.
2. **Constraint Check** — If the choice violates a constraint, **prune** (abandon this path).
3. **Recurse** — If valid, move to the next decision point.
4. **Backtrack** — If a dead end is reached, undo the last choice and try the next option.

### 2.2.1 Algorithm: `solveNQueens(col)`

```
function solve(col):
    if col >= N:
        record current board as a SOLUTION
        return

    for row = 0 to N-1:
        if isSafe(row, col):
            place queen at board[row][col]
            solve(col + 1)              // recurse for next column
            remove queen from board[row][col]  // BACKTRACK
```

### 2.2.2 Safety Check: `isSafe(row, col)`

Before placing a queen at position `(row, col)`, check three directions **to the left** (since queens are placed column by column, left to right):

| Direction | Check |
|-----------|-------|
| **Same Row (left)** | Iterate `j` from `0` to `col-1`; if `board[row][j] == 1`, conflict. |
| **Upper-Left Diagonal** | Iterate `(i, j)` starting from `(row, col)` going `(i-1, j-1)` until out of bounds. |
| **Lower-Left Diagonal** | Iterate `(i, j)` starting from `(row, col)` going `(i+1, j-1)` until out of bounds. |

> **Why only check left?** Because queens are placed one per column from left to right, so no queen can exist to the right of the current column yet.

### 2.2.3 State-Space Tree

The algorithm explores a **state-space tree** where:
- **Level `k`** = placing a queen in column `k`.
- **Branching factor** = up to `N` (one branch per row in that column).
- **Pruning** removes branches where a queen placement causes a conflict.

```
                        Root (col=0)
                     /    |    \    \
                 row 0  row 1  row 2  row 3     ← Column 0 choices
                 / | \   / | \   ...
              r0 r1 r2  r0 r1 r2  ...           ← Column 1 choices
              ✗  ✗  ✓   ...                     ← Pruned / Valid
```

# Chapter 3: Code (Commented) with Intermediate Steps

## 3.1 Core Algorithm — `generateSteps(N)` in `SimulatorTab.jsx`

This is the heart of the application. It generates the complete algorithm trace **before** the simulation begins, so the user can step through it at their own pace.

```js
// generateSteps: Pre-computes every step of the backtracking algorithm
// Input:  N (board size)
// Output: { steps: Step[], solutions: number[][][] }
const generateSteps = (N) => {
    // STEP 1: Initialize an empty N×N board (all zeros)
    let internalBoard = Array(N)
      .fill(0)
      .map(() => Array(N).fill(0));

    let localSteps = [];          // Array to record every algorithm action
    const solutions = [];         // Array to store all valid final boards

    // ------------------------------------------------------------------
    // HELPER: getConflicts(r, c)
    // Checks if placing a queen at (r, c) conflicts with any existing queen.
    // Only checks LEFT of column c (since we place left-to-right).
    // Returns an array of conflicting queen positions.
    // ------------------------------------------------------------------
    const getConflicts = (r, c) => {
      let conflicts = [];

      // Check 1: Same row, all columns to the left
      for (let i = 0; i < c; i++) {
        if (internalBoard[r][i] === 1)
          conflicts.push({ row: r, col: i });
      }

      // Check 2: Upper-left diagonal
      for (let i = r, j = c; i >= 0 && j >= 0; i--, j--) {
        if (internalBoard[i][j] === 1)
          conflicts.push({ row: i, col: j });
      }

      // Check 3: Lower-left diagonal
      for (let i = r, j = c; j >= 0 && i < N; i++, j--) {
        if (internalBoard[i][j] === 1)
          conflicts.push({ row: i, col: j });
      }

      return conflicts;
    };

    // ------------------------------------------------------------------
    // HELPER: clone()
    // Creates a deep copy of the current board state.
    // This is critical — without cloning, all steps would reference
    // the same (mutated) board object.
    // ------------------------------------------------------------------
    const clone = () => internalBoard.map((row) => [...row]);

    // ------------------------------------------------------------------
    // RECURSIVE FUNCTION: solve(col)
    // Attempts to place a queen in each row of the given column.
    // If successful, recurses to the next column.
    // If no row works, returns (triggering backtracking).
    // ------------------------------------------------------------------
    const solve = (col) => {
      // BASE CASE: All N columns filled → solution found!
      if (col >= N) {
        localSteps.push({
          type: "solution",
          board: clone(),
          log: {
            title: `Solution ${solutions.length + 1} Found!`,
            details: "All queens successfully placed without threatening each other.",
            bg: "#dcfce7",       // Green background
            border: "#22c55e",   // Green border
          },
        });
        solutions.push(clone());  // Store this valid configuration
        return false;             // Return false to continue searching for MORE solutions
      }

      // RECURSIVE CASE: Try each row in the current column
      for (let row = 0; row < N; row++) {

        // Step A: Record that we are TESTING this cell
        localSteps.push({
          type: "placing",
          row, col,
          board: clone(),
          log: {
            title: `Testing [Row ${row}, Col ${col}]`,
            details: "Checking if a Queen can be safely placed here.",
            bg: "#fef9c3",       // Yellow background
            border: "#eab308",
          },
        });

        // Step B: Check for conflicts
        const conflicts = getConflicts(row, col);

        if (conflicts.length === 0) {
          // --- SAFE: Place the queen ---
          internalBoard[row][col] = 1;

          localSteps.push({
            type: "placed_safe",
            row, col,
            board: clone(),
            log: {
              title: `Placed Queen at [Row ${row}, Col ${col}]`,
              details: "Safe position. Moving to next column.",
              bg: "#e0f2fe",     // Blue background
              border: "#38bdf8",
            },
          });

          // Step C: Recurse to the next column
          solve(col + 1);

          // Step D: BACKTRACK — Remove the queen and try the next row
          internalBoard[row][col] = 0;

          localSteps.push({
            type: "backtrack",
            row, col,
            board: clone(),
            log: {
              title: `Backtracking from [Col ${col}]`,
              details: `Dead end reached in subsequent columns. Removing Queen at [Row ${row}, Col ${col}] and trying next row.`,
              bg: "#fef2f2",     // Red background
              border: "#ef4444",
            },
          });

        } else {
          // --- CLASH: Cannot place here ---
          localSteps.push({
            type: "clash",
            row, col,
            conflicts,          // Store which queens cause the conflict
            board: clone(),
            log: {
              title: `Clash at [Row ${row}, Col ${col}]`,
              details: "Threatened by existing Queen(s). Cannot place here.",
              bg: "#fee2e2",
              border: "#f87171",
            },
          });
        }
      }

      return false;  // No valid row found in this column → backtrack
    };

    // Start the algorithm from column 0
    solve(0);

    return { steps: localSteps, solutions };
};
```

### Intermediate Steps Breakdown

For a **4×4 board**, the algorithm generates the following sequence of step types:

| Step # | Type | Position | What Happens |
|--------|------|----------|--------------|
| 1 | `placing` | (0,0) | Testing row 0, col 0 |
| 2 | `placed_safe` | (0,0) | No conflicts → queen placed |
| 3 | `placing` | (0,1) | Testing row 0, col 1 |
| 4 | `clash` | (0,1) | Conflict with queen at (0,0) — same row |
| 5 | `placing` | (1,1) | Testing row 1, col 1 |
| 6 | `clash` | (1,1) | Conflict with queen at (0,0) — diagonal |
| 7 | `placing` | (2,1) | Testing row 2, col 1 |
| 8 | `placed_safe` | (2,1) | Safe → queen placed |
| 9 | `placing` | (0,2) | Testing row 0, col 2 |
| ... | ... | ... | ... |
| k | `solution` | — | All 4 queens placed! Solution found. |
| ... | `backtrack` | various | Continue searching for more solutions. |

## 3.2 Board Cloning — Immutable Snapshots

```js
// Deep clone using map + spread operator
const clone = () => internalBoard.map((row) => [...row]);
```

**Why this matters:** JavaScript arrays are reference types. Without cloning, every step object would point to the **same** board array, which gets mutated. The spread operator `[...row]` creates a **shallow copy** of each row, which is sufficient because the elements are primitive numbers (0 or 1).

## 3.5 Cell Colouring Logic — `getCellColor(i, j)`

```js
const getCellColor = (i, j) => {
    const isDark = (i + j) % 2 === 1;
    const baseColor = isDark ? "#cbd5e1" : "#f8fafc";  // Alternating chess pattern

    if (!currentStepData) return baseColor;

    const { type, row, col, conflicts } = currentStepData;

    // Highlight the cell being acted upon
    if (i === row && j === col) {
      if (type === "clash")                       return "#fca5a5";  // Red
      if (type === "placed_safe" || type === "solution") return "#86efac";  // Green
      if (type === "placing")                     return "#fef08a";  // Yellow
      if (type === "backtrack")                   return "#fca5a5";  // Red
    }

    // Highlight conflicting queens in red
    if (type === "clash" && conflicts?.some(c => c.row === i && c.col === j)) {
      return "#fca5a5";  // Red
    }

    return baseColor;
};
```

# Chapter 5: Complexity Analysis

## 5.1 Time Complexity

### Worst Case: O(N!)

| Aspect | Analysis |
|--------|----------|
| **Column 0** | N choices (try each row) |
| **Column 1** | At most N−1 valid choices (one row eliminated by column 0's queen) |
| **Column 2** | At most N−2 valid choices |
| **...** | ... |
| **Column N−1** | At most 1 valid choice |

**Total nodes explored ≤ N × (N−1) × (N−2) × ... × 1 = N!**

However, **pruning** significantly reduces this in practice. The `getConflicts()` check eliminates branches early. The **effective branching factor** is much less than N at deeper levels.

### Comparison with Brute Force

| Approach | Time Complexity | Explanation |
|----------|-----------------|-------------|
| **Brute Force** (try all placements) | O(Nᴺ) | Place a queen in any row of each column, check all combos |
| **Backtracking** (this project) | O(N!) | Prune invalid branches immediately |
| **Optimised Backtracking** (bitmasking) | O(N!) but faster constant | Use bitmasks to check safety in O(1) |

### Empirical Step Counts

| Board Size (N) | Total Steps Generated | Solutions Found |
|----------------|-----------------------|-----------------|
| 4 | ~60 | 2 |
| 5 | ~300 | 10 |
| 6 | ~2,000 | 4 |
| 7 | ~12,000 | 40 |
| 8 | ~90,000 | 92 |

## 5.2 Space Complexity

| Component | Space | Explanation |
|-----------|-------|-------------|
| **Board** (`internalBoard`) | O(N²) | N×N 2D array |
| **Recursion Stack** | O(N) | Maximum depth = N (one call per column) |
| **Steps Array** | O(S × N²) | S steps, each storing an N×N board clone |
| **Solutions Array** | O(K × N²) | K solutions, each an N×N board |

Where S = total number of steps and K = number of valid solutions.

> **Note:** The `steps` array is the dominant memory consumer because each step stores a complete board snapshot. For N=8, this can be ~90,000 × 64 = ~5.76 million integers. This is acceptable for browser memory but is why the app limits N to 10.

## 5.3 Safety Check Complexity — `getConflicts()`

```
Row check:        O(N)     — scan one row left of current column
Upper diagonal:   O(min(row, col))  — at most N iterations
Lower diagonal:   O(min(N-row, col))  — at most N iterations
─────────────────────────
Total per call:   O(N)
```

This is called once per cell test, and there are up to N! cell tests, giving an overall conflict-checking cost of **O(N × N!) = O(N · N!)** in the worst case.

# Chapter 6: Conclusion

## 6.1 Summary

This virtual lab project successfully demonstrates the **N-Queen Problem** using a **backtracking** algorithm, implemented as an interactive **React** web application. The key achievements are:

1. **Algorithm Implementation** — A complete backtracking solver that finds **all** valid solutions for any board size N (4 ≤ N ≤ 10).

2. **Step-by-Step Visualisation** — Every algorithmic action (test, place, clash, backtrack, solution) is recorded and can be stepped through by the user, making the backtracking process transparent and educational.

3. **Interactive UI** — The application features a sidebar for navigation, a theory section explaining the algorithm, and a simulator with a chessboard grid and a real-time log panel.

4. **Modern Web Technologies** — Built with **React 19** (hooks-based functional components), **Vite 8** (instant dev server and optimised builds), and **ES6+ JavaScript** (arrow functions, destructuring, spread operators, template literals).

## 6.2 Key Learnings

| Area | What We Learned |
|------|-----------------|
| **Backtracking** | How to systematically explore a search space and prune invalid branches |
| **React State Management** | Using `useState`, `useEffect`, and `useRef` for complex interactive simulations |
| **Component Architecture** | Breaking a complex UI into reusable, focused components with clear prop interfaces |
| **Algorithm Visualisation** | Pre-computing algorithm traces and replaying them step-by-step for educational value |
| **Immutability** | The importance of cloning state (board snapshots) to preserve history |

## 6.3 Future Enhancements

- **Auto-play mode** — Automatically advance steps with adjustable speed.
- **Bitmasking optimisation** — Use bitmasks for O(1) safety checks instead of O(N) loops.
- **Performance mode** — Skip board cloning and only record metadata for large N values.
